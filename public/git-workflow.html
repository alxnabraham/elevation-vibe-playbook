Git + Vercel Workflow Guide

Everything you need to make changes, preview them on your own URL, and get them into the live demo site. This guide assumes zero coding knowledge.

Before you start
This guide assumes you already have VS Code installed and a GitHub account connected to the Elevation AI organization. If you don't have these, message Alan on Slack first.

1. The Big Picture

Before diving into steps, here is a plain-English explanation of how our system works.

What are Git, GitHub, and Vercel?

Git is a tool that tracks every change anyone makes to our code. Think of it like Google Docs version history, but for code.

GitHub is the website where our code lives online. It is like a shared Google Drive folder, but specifically for code. Our project is stored there under the Elevation AI organization.

Vercel is the service that takes our code and turns it into a live website you can visit in your browser. Every time code is pushed to GitHub, Vercel automatically rebuilds the website with the latest changes.

How changes flow

Here is the journey every change takes, from your computer to the live website:

You edit code in VS Code
Push to your branch on GitHub
Vercel auto-builds your preview URL
You see it live!

Our branch structure

Think of branches as separate copies of the project. Each person works on their own copy so nobody steps on anyone else's work.

Branch NamePurposeWho Touches It
demoThe version shown to investors and clients. This is the live product.Nobody directly. Alan merges approved changes here.
stagingThe testing ground. Changes go here first before reaching demo.Alan merges reviewed changes here.
dev-[yourname]Your personal workspace. This is where you make all your changes.Only you. Example: dev-sachin, dev-kevin, dev-chris.
The Golden Rule
NEVER push directly to the demo or staging branches. Always work on your own dev-[yourname] branch. When your changes are ready, Alan or the tech lead will merge them.

Step B — Clone the project to your computer

This downloads a copy of the entire project to your computer so you can work on it.

Open VS Code.

Press Ctrl + Shift + P (Windows) or Cmd + Shift + P (Mac). This opens the Command Palette.

Type Git: Clone and select it from the dropdown.

It will ask for a URL. Paste the repo URL that Alan shared in the Slack #ws-apps channel.

Choose a folder on your computer where you want to save it (e.g., your Documents or Desktop).

VS Code will download the project and ask if you want to open it. Click Open.

Using Cursor or Windsurf instead of VS Code?
The steps are identical. These tools are built on VS Code, so all commands work the same way.

Step C — Switch to your personal branch

After cloning, you will be on the demo branch by default. You need to switch to your personal branch.

Look at the bottom-left corner of VS Code. You will see a branch name (probably "demo" or "main").

Click on that branch name. A dropdown will appear at the top of the screen.

Find and select your branch: dev-[yourname] (e.g., dev-sachin, dev-kevin, dev-chris, dev-hanzel).

If you do not see your branch, select "Create new branch from..." and type your branch name, then select demo as the source.

How to verify you are on the right branch
Look at the bottom-left corner of VS Code. It should show your branch name (e.g., dev-sachin). If it says demo or staging, you are on the wrong branch.

Step D — Set up environment variables

The project needs some secret keys to connect to our backend services. Without these, the app will show "Access Denied."

In Slack, go to the #ws-apps channel. Find the pinned message with the environment variables.

In VS Code, find the project root folder (the top level). Create a new file called exactly .env.local (note the dot at the beginning).

Copy and paste the entire contents from the Slack message into this file.

Save the file.

Important
Never share this file or commit it to GitHub. It contains private keys. The project is already set up to ignore this file automatically, but do not rename it or move it.

Step E — Auth and cookie setup

Our app uses authentication that requires a small one-time cookie setup to work with your Vercel preview URL.

Open your browser and go to dev-agents.elevation.com.

Sign in with your @elevationai.com Google account.

Once signed in, right-click anywhere on the page and select Inspect (or press F12).

In the panel that opens, click the Application tab at the top.

In the left sidebar, expand Cookies and click on the current site.

Find the cookies named elevation_dev and dev_user. For each one, double-click the Domain column and change it to your Vercel preview URL domain (the part after https://, without the https://). Alan will share your specific URL.

Refresh the page. You should now see the app.

3. Your Daily Workflow

This is the section you will use every day. The steps below are what you do each time you sit down to work on the app.

EVERY TIME you open VS Code to work, start with Step 1
Skipping Step 1 means you are working on old code. This causes merge conflicts that are painful to fix and can break other people's work.

Step 1 — Sync your branch (pull latest changes)

Before you touch anything, pull the latest code so you are working with the most recent version.

Open VS Code and confirm you are on your branch (check bottom-left corner — it should say dev-[yourname]).

Open the Terminal in VS Code: go to the top menu bar and click Terminal → New Terminal.

In the terminal, type the following and press Enter:

git pull origin staging

This downloads the latest approved code from the staging branch into your branch.

If there are no errors, you are ready to work. If you see a "merge conflict" error, stop and message Alan on Slack.

What does "git pull origin staging" mean?
git pull = download latest changes. origin = from GitHub. staging = from the staging branch (the source of truth for development).

Step 2 — Make your changes

Now you can edit files, work with your AI coding assistant, or make whatever changes you need.

  • Use the file explorer on the left side of VS Code to navigate to the files you want to edit.
  • If you are using an AI assistant (Claude, Cursor, Windsurf), you can ask it to make changes and it will edit the files for you.
  • Save your files frequently: Ctrl+S (Windows) or Cmd+S (Mac).

For detailed guidance on working with AI coding assistants, see the AI-Assisted Development Guide.

Step 3 — Push your changes to GitHub

When you are happy with your changes, push them to GitHub so Vercel can build your preview.

Option A: Using VS Code's Git panel (recommended for beginners)

Click the Source Control icon in the left sidebar (it looks like a branching line, or press Ctrl+Shift+G / Cmd+Shift+G).

You will see a list of all files you changed. Review them.

In the text box at the top that says "Message", type a short description of what you changed. Example: "Fixed dashboard background color" or "Updated calendar layout".

Click the checkmark button to commit. If asked whether to stage all changes, click Yes.

Click the three dots (...) menu at the top of the Source Control panel, then click Push.

Option B: Using the terminal (faster once you are comfortable)

Open the terminal in VS Code and run these three commands one by one:

git add .
git commit -m "describe your change here"
git push

Step 4 — Check your preview URL

After pushing, Vercel automatically rebuilds your preview. This usually takes 1–3 minutes.

Go to vercel.com and open the Elevation AI team.

Click on the EI V2 Redesign project.

You will see Active Branches listed. Find your branch name (dev-[yourname]) and click its preview URL.

Your latest changes should appear. If Vercel says "Building", wait a minute and refresh.

Your personal preview URL
Each person has their own URL that only shows code from their branch. Alan will share your specific URL. Bookmark it — you will visit it often.

4. Getting Your Changes Into Demo

Once you are happy with how your changes look on your preview URL, here is how they get to the live demo site.

Message Alan or the tech lead on Slack: "My changes on dev-[yourname] are ready to merge. Here is what I changed: [brief description]."

Alan will review your changes on your preview URL and on GitHub.

If approved, Alan will create a Pull Request (PR) from your branch to staging first for a final check, then to demo.

Once merged to demo, your changes are live on the investor-facing site.

Do NOT merge your own Pull Requests
Even if GitHub lets you click the Merge button, do not do it. All merges to staging and demo go through Alan or the designated tech lead to prevent breaking the live site.

5. Troubleshooting

Here are the most common problems and exactly how to fix them. For a complete list, see the Troubleshooting Guide.

"Access Denied" on the app

  • Make sure you are signed in at dev-agents.elevation.com with your @elevationai.com account.
  • Check that your cookies are set correctly (see Section 2, Step E).
  • Try opening the site in an Incognito / Private window and signing in fresh.
  • Clear your browser cookies for the site and redo the cookie setup.
  • If none of this works, message Alan. Jay may need to add your preview URL to the backend.

"Merge conflict" error when pulling

This means someone else changed the same file you changed. Do not try to fix this yourself.

  • Take a screenshot of the error.
  • Message Alan on Slack with the screenshot.
  • Alan will help resolve the conflict or do it for you.

"Does not have contributing access" on Vercel

This means your Vercel account is not properly linked to the Elevation AI team.

  • Go to vercel.com and check if you see "Elevation AI" in the top-left team dropdown.
  • If you do not see it, message Alan to resend the team invite.
  • Accept the invite from your email and try pushing again.

My preview URL shows old content

  • Wait 2–3 minutes. Vercel takes time to rebuild after each push.
  • Hard-refresh your browser: Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac).
  • Check Vercel's dashboard to see if the build is still running or if it failed.

I accidentally pushed to the wrong branch

Do not panic. Message Alan immediately with which branch you pushed to and what changes you made. This is fixable.
