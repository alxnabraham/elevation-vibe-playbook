Vibe Coding Playbook

Clone, code with AI, push, preview, ship. This is everything you need to start building on the EI V2 platform.

1. Clone & Install (one-time)

Open VS Code (or Cursor / Windsurf — they all work the same).

Clone the repo. Press Cmd+Shift+P (Mac) or Ctrl+Shift+P (Windows), type Git: Clone, and paste:

https://github.com/Elevation-AI/ei-v2-redesign.git

Install dependencies. Open the terminal (Terminal → New Terminal) and run:

npm install

Set up environment variables. In Slack #ws-apps, find the pinned .env.local message. Create a file called .env.local in the project root and paste the contents.

Never commit .env.local
This file contains private keys. It is already git-ignored. Do not rename, move, or share it.

2. Download the Agent File (one-time)

This file teaches your AI assistant (Copilot, Cursor, etc.) our framework rules so it generates correct code from the start.

Download the agent file:

https://docs.elevationai.com/vibe/assets/vibeapps.agent.md

Open that URL, select all (Cmd+A), and copy (Cmd+C).

Create the file in your project. In VS Code, create the folder .github/agents/, then create a file called vibeapps.agent.md inside it. Paste the contents and save.

Verify. Open Copilot Chat (Cmd+Option+I), set mode to Agent, and you should see elevationai-app in the agent picker.

Using Claude as your model
In the Copilot model picker, select Claude Sonnet 4.5 or Claude Sonnet 4.6 for the best results. Or leave it on Auto.

4. Your Daily Workflow — The 5-Step Cycle

Every time you sit down to work, follow these 5 steps in order. This is the most important section of this entire guide.

1. Pull latest
2. Code with AI
3. Push to your branch
4. Check preview
5. Push to sandbox

Step 1 — Pull the latest code

Do this EVERY TIME before you start — no exceptions
If you skip this, your AI assistant will work on stale code and create merge conflicts that break things for everyone.

Confirm you are on your branch (check bottom-left of VS Code — it should say dev-[yourname]).

Open the terminal and run:

git pull origin staging

This downloads the latest code from the shared staging branch into your branch.

If you see "Already up to date" — you are good. If you see "merge conflict" — stop and message Alan.

Step 2 — Code with your AI assistant

Now you can start building. Use Copilot, Cursor, Claude Code, or Windsurf to make your changes.

AI assistants sometimes change files you didn't ask about
Before moving to Step 3, open the Source Control panel (Cmd+Shift+G) and review every file that was modified. If the AI changed package.json, config files, or files unrelated to your task — right-click those files and select Discard Changes.

Step 3 — Push to your branch

Your changes only exist on your computer until you push. If you close VS Code without pushing, nothing gets saved to GitHub or deployed to your preview URL.

Option A: Tell your AI assistant (easiest)

Type this in your AI chat:

Push all my changes to my branch on GitHub with the message: "describe what you changed"

Option B: Use the VS Code panel

Open Source Control (Cmd+Shift+G).

Type a short message describing your change (e.g., "Fixed dashboard layout").

Click the checkmark to commit. If asked to stage all changes, click Yes.

Click ... → Push.

Option C: Terminal commands

git add .
git commit -m "describe your change here"
git push

Step 4 — Check your preview URL

After pushing, Vercel automatically rebuilds your preview in 1–3 minutes.

Open your preview URL (see the table above — bookmark it).

Hard-refresh: Cmd+Shift+R (Mac) or Ctrl+Shift+R (Windows).

Verify your changes look correct. Click around, test the pages you modified.

Do NOT skip this step
If your preview looks broken, fix it before pushing to sandbox. Once it is in sandbox, a broken change affects the shared environment.

Step 5 — Push to sandbox

Once your preview looks good, merge your work into the shared sandbox branch.

Message Alan on Slack in #ws-apps:

"My changes on dev-[yourname] are ready. I've verified on my preview URL. Here's what I changed: [brief description]."

Alan will review and merge your branch into sandbox.

Never push directly to sandbox or demo
All merges to sandbox and demo go through Alan. This prevents breaking the shared preview and the investor-facing site.

Think of branches as separate copies of the project. Each person works on their own copy so nobody steps on anyone else's work.
